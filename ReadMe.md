# Canister Worm


## معرفی 
بدافزار CanisterWorm که با نام فنی TSCP-26 نیز شناخته می‌شود، پیچیده‌ترین حمله زنجیره تأمین (Supply Chain Attack) تا به امروز است. این کرم اولین بار در ۱ مارس ۲۰۲۶ شناسایی شد و به دلیل استفاده از زیرساخت‌های بلاک‌چینی برای فرماندهی، فصلی جدید در تاریخ بدافزارها باز کرد.

این بدافزار یک کرم هیبریدی (Hybrid Worm) است که با هدف قرار دادن پکیج‌های توسعه‌دهندگان در اکوسیستم npm، به دنبال سرقت دارایی‌های ابری و تخریب زیرساخت‌های حیاتی است. نام آن از کلمه Canister (واحدهای پردازشی در شبکه بلاک‌چین ICP) گرفته شده است، چرا که مغز متفکر این بدافزار نه روی یک سرور معمولی، بلکه روی یک قرارداد هوشمند غیرمتمرکز میزبانی می‌شود.

چه کسی و چطوری آن را پیدا کرد؟
کشف این بدافزار نه توسط یک آنتی‌ویروس، بلکه توسط یک مهندس ارشد امنیت در شرکت Aqua Security به نام "الکساندر رومانوف" انجام شد. داستان کشف به شرح زیر است:

ناهنجاری در ترافیک شبکه (نقطه شروع):
در اواخر فوریه ۲۰۲۶، رومانوف متوجه شد که یکی از سرورهای آزمایشی آن‌ها که ابزار Trivy (اسکنر امنیتی معروف) روی آن نصب بود، ترافیک عجیبی به سمت دامنه‌های .icp0.io ارسال می‌کند. این دامنه‌ها مربوط به شبکه "اینترنت کامپیوتر" هستند که معمولاً در محیط‌های توسعه نرم‌افزار دلیلی برای ارتباط با آن‌ها وجود ندارد.

کشف دستکاری در کد منبع (The Tweak):
تیم امنیتی متوجه شد که نسخه نصب شده از یک پکیج داخلی به نام @trivy-core/plugin-syslog دارای ۲ خط کد اضافه در فایل index.js است. این کدها هیچ اثری در مخزن (Repository) اصلی گیت‌هاب نداشتند، اما در نسخه منتشر شده در سایت npm موجود بودند. این نشان‌دهنده یک حمله Publish-only بود (هکرها فقط نسخه منتشر شده را آلوده کرده بودند).

تحلیل معکوس و کشف "زنجیره سمی":
با بررسی عمیق‌تر، مشخص شد که هکرها (گروه TeamPCP) با سرقت توکن انتشارِ یکی از توسعه‌دهندگان اصلی Trivy، توانسته‌ بودند کد مخرب را تزریق کنند. آن‌ها نه تنها Trivy را آلوده کرده بودند، بلکه بدافزار به محض نصب روی سیستم هر توسعه‌دهنده دیگر، توکن‌های npm او را می‌دزدید و به طور خودکار پکیج‌های او را هم آلوده می‌کرد.

کشف ماهیت "کرم" (Worm Capability):
رومانوف مشاهده کرد که این بدافزار به صورت خودکار در کلاستر کوبرنتیز شرکت در حال پخش شدن است. بدافزار با استفاده از دسترسی‌های CI/CD، خودش را به شکل یک DaemonSet پنهان به نام kube-proxy-monitor بازنویسی کرده بود تا در کل زیرساخت تکثیر شود.

چه کسی پشت این حمله است؟
تحقیقات شرکت‌های Mandiant و CrowdStrike نشان داد که ردپای دیجیتال بدافزار به گروهی به نام TeamPCP می‌رسد. این گروه با استفاده از هوش مصنوعی مولد برای نوشتن کدهای پولی‌مورفیک (چندشکلی)، موفق شده بود بیش از ۱۴۰۰ پکیج npm را در کمتر از ۴۸ ساعت آلوده کند، پیش از آنکه اولین زنگ خطر توسط رومانوف به صدا درآید.

## بررسی رفتار های کرم
این حمله از یک مدل «آبشاری» (Cascading) پیروی می‌کند که در آن، هر نفوذ موفق به عنوان سوخت و ابزاری برای نفوذ بعدی استفاده می‌شود. این جریان با رخنه به ابزار Trivy (اسکنر امنیتی محبوب شرکت Aqua Security) آغاز شد که به مهاجمان اجازه داد توکن‌های GitHub و npm را از سیستم‌های توسعه‌دهندگان و خط لوله‌های CI/CD سرقت کنند.

نقطه ورود (Entry Point):
کد مخرب در یک اسکریپت postinstall درون یک پکیج npm آلوده پنهان شده است. به محض اینکه توسعه‌دهنده دستور npm install را اجرا می‌کند، این اسکریپت به صورت خودکار و بدون نیاز به تایید، در پس‌زمینه اجرا می‌شود.

برداشت اعتبارنامه‌ها (Credential Harvesting):
بدافزار بلافاصله سیستم را برای یافتن مقادیر _authToken در فایل‌های .npmrc مربوط به تنظیمات npm، متغیرهای محیطی (مانند NPM_TOKEN) و سایر پیکربندی‌های سراسری سیستم اسکن می‌کند.

خود-تکثیری یا حالت کرم (The "Worm"):
بدافزار با استفاده از توکن‌های سرقت شده، از رجیستری npm استعلام می‌گیرد تا ببیند قربانی اجازه انتشار چه پکیج‌های دیگری را دارد. سپس به طور خودکار نسخه (Version) آن پکیج‌ها را بالا برده و یک نسخه آلوده (Backdoored) از آن‌ها را آپلود می‌کند. با این کار، آلودگی به تمام کاربران آن پکیج‌های جدید نیز سرایت می‌کند.

فرماندهی و کنترل غیرمتمرکز (Decentralized C2):
بدافزار یک بک‌دور (Backdoor) مبتنی بر پایتون نصب می‌کند که به طور مداوم یک کانستر ICP (نوعی قرارداد هوشمند در پروتکل Internet Computer) را چک می‌کند. از آنجایی که سرور فرماندهی (C2) روی یک بلاک‌چین میزبانی می‌شود، مسدود کردن یا از کار انداختن آن توسط مقامات امنیتی (برخلاف سرورهای وب معمولی) بسیار دشوار و تقریباً غیرممکن است.

فعال‌سازی از طریق npm Hook:
بدافزار در فایل package.json پکیج آلوده، در قسمت "scripts"، از دستور "postinstall": "node .lib/init.js" استفاده می‌کند. به محض پایان نصب پکیج، این اسکریپت بدون دخالت کاربر اجرا می‌شود.

استقرار پی‌لود پایتونی (Shadow Payload):
اسکریپت init.js یک فایل پایتون را که به صورت هگزادسیمال (Hex) در دل خود پنهان کرده، در مسیر 

    ~/.local/share/pgmon/service.py
    
 بازنویسی می‌کند. این مسیر معمولاً توسط اسکنرهای امنیتی عادی بررسی نمی‌شود.

ایجاد پایداری (Persistence) با استفاده از systemd:
بدافزار یک فایل Unit برای systemd در مسیر زیر ایجاد می‌کند:

    ~/.config/systemd/user/pgmon.service

محتوای این فایل به گونه‌ای تنظیم شده که بدافزار را به یک "سرویس رویین‌تن" تبدیل کند:

Restart=always:

 اگر ادمین پروسه را بکشد، در کمتر از ۵ ثانیه دوباره زنده می‌شود.

ExecStart:

 مستقیماً مفسر پایتون را روی اسکریپت مخفی شده اجرا می‌کند.

سرقت توکن‌های محلی:
به محض اجرا، سرویس pgmon فایل‌های .npmrc و .ssh/id_rsa و همچنین متغیرهای محیطی (Environment Variables) را برای یافتن کلیدهای دسترسی به AWS و GitHub اسکن کرده و به کانستر ICP ارسال می‌کند.

## کرم بعد از نفوذ به سیستم قربانی چیکار میکنه 
بخش ۱: تثبیت پایداری از طریق systemd (سرویس نامرئی)

* زمانی که یک توسعه‌دهنده یا سرورِ بیلد (Build Server) دستور npm install را اجرا می‌کند، هوکِ postinstall در پکیج آلوده (مانند @emilgroup/form-builder) یک لودر Node.js را اجرا می‌کند. این لودر مسئول کاشتن بک‌دور (Backdoor) دائمی در سیستم است.

* استقرار فایل مخرب (Payload Dropping):
لودر یک رشته طولانی Base64 را که در کدهای جاوااسکریپت جاسازی شده، رمزگشایی می‌کند و آن را به صورت یک اسکریپت پایتون در یک مسیر محلی مخفی می‌نویسد:

مسیر: ~/.local/share/pgmon/service.py

پوشش: از نام pgmon (مخفف PostgreSQL Monitor) استفاده می‌کند تا با ابزارهای قانونی مدیریت دیتابیس ترکیب شده و شناسایی نشود.

------------------
۲. ساخت سرویس (بدون نیاز به دسترسی Root):

* بدافزار یک سرویس systemd در سطح کاربر (User-level) ایجاد می‌کند. برخلاف سرویس‌های سراسری سیستم، سرویس‌های کاربر در دایرکتوری Home قرار می‌گیرند و برای نصب یا اجرا نیازی به امتیازات sudo یا root ندارند.

فایل ایجاد شده:

     ~/.config/systemd/user/pgmon.service

پیکربندی: پارامترهای Restart=always و RestartSec=5 تنظیم می‌شوند. این کار تضمین می‌کند که حتی اگر پروسه پایتون کرش کند یا به صورت دستی کشته شود، systemd به طور خودکار ۵ ثانیه بعد آن را احیا می‌کند.

۳. فعال‌سازی فوری:
لودر سه دستور را در پس‌زمینه اجرا می‌کند:

    systemctl --user daemon-reload (برای ثبت سرویس جدید).

    systemctl --user enable pgmon.service (برای اطمینان از اجرا در هر بار بوت).

    systemctl --user start pgmon.service (برای شروع فوری ارتباط با سرور فرماندهی).


---------------
بخش ۲: تصاحب کلاستر کوبرنتیز (Escalation to K8s)

بک‌دور پایتونی (service.py) به عنوان یک ارکستراتور عمل می‌کند. به محض اجرا، شروع به اثرانگشت‌برداری از محیط (Environment Fingerprinting) می‌کند. اگر تشخیص دهد که داخل یک کانتینر یا روی ماشینی با دسترسی kubectl است، ماژول بهره‌برداری از کوبرنتیز را فعال می‌کند.

۱. شناسایی پاد و سرقت اعتبارنامه‌ها:

بدافزار وجود توکنِ "سرویس اکانت" را بررسی می‌کند:

بررسی مسیر: 

    /var/run/secrets/kubernetes.io/serviceaccount/token

بررسی متغیر محیطی: 

    KUBERNETES_SERVICE_HOST

اگر هر یک موجود باشد، از آن توکن برای احراز هویت در API کوبرنتیز استفاده می‌کند.


------------
۲. حرکت عرضی (Lateral Movement) از طریق DaemonSet:

اگر سرویس‌اکانت سرقت شده اجازه ساخت منابع را داشته باشد (که در CI/CD Runnerها رایج است)، بدافزار یک DaemonSet با امتیازات ویژه (معمولاً به نام host-provisioner-std) مستقر می‌کند.


پوشش کامل:
 از آنجایی که این یک DaemonSet است، کوبرنتیز به طور خودکار یک پاد مخرب را روی تک‌تک نودهای کلاستر برنامه‌ریزی و اجرا می‌کند.

فرار از محدودیت‌ها: 
بدافزار شامل تنظیمات toleration برای operator: Exists است که به آن اجازه می‌دهد از سدّ Taintها عبور کرده و حتی نودهای Master را آلوده کند.

-----------------------
۳. فرار از کانتینر (Host Escape):
پادهای این DaemonSet با امتیازات افراطی پیکربندی شده‌اند:

    privileged: true: 

به کانتینر دسترسی تقریباً کامل به کرنل (هسته) میزبان می‌دهد.

    hostPID: true:

 به کانتینر اجازه می‌دهد تمامی پروسه‌های در حال اجرا روی نود میزبان را مشاهده کرده و با آن‌ها تعامل کند.

Mount کردن میزبان: 

فایل‌سیستم ریشه نود (/) را در مسیری مانند /mnt/host داخل کانتینر مونت می‌کند.

۴. آلودگی بازگشتی:

به محض اینکه پادهای DaemonSet روی هر نود اجرا شدند، اسکریپتی را برای نصب بک‌دور systemd (که در بخش ۱ توضیح داده شد) مستقیماً روی سیستم‌عامل اصلی (Host OS) هر نود اجرا می‌کنند.

نتیجه: 

حتی اگر کل کلاستر کوبرنتیز را حذف کنید، بدافزار اکنون به صورت فیزیکی روی ماشین‌های مجازی یا سرورهای زیرین نصب شده است.

-----------------
بخش ۳: 

منطق تخریبی "کامیکازه" (Wiper Logic)
در اواخر مارس ۲۰۲۶، یک نسخه مخرب از این بدافزار مشاهده شد. قبل از برقراری بک‌دور، بدافزار منطقه زمانی (Timezone) و زبان (Locale) سیستم را چک می‌کند.

بررسی: اگر منطقه زمانی روی Asia/Tehran یا زبان روی fa_IR (فارسی) تنظیم شده باشد:

عملیات: به جای نصب بک‌دور، DaemonSet یک دستور پاک‌کننده (Wiper) اجرا می‌کند:

    rm -rf /mnt/host/* --no-preserve-root && reboot -f

نتیجه: این کار عملاً تمام نودهای کلاستر را به طور هم‌زمان نابود (Brick) می‌کند و منجر به از دست رفتن کامل داده‌ها و فروپاشی زیرساخت در سیستم‌های هدف قرار گرفته در ایران می‌شود.

## چطور ببینیم سیستم مون آلوده شده یا نه

1-  بررسی فایل‌های مخرب روی سیستم

 بررسی وجود Python Implant (مهم‌ترین نشانه آلودگی)

    ls -la ~/.local/share/pgmon/service.py

### بررسی systemd persistence unit

    ls -la ~/.config/systemd/user/pgmon.serv

### بررسی second-stage payload

    ls -la /tmp/pglog

### بررسی download state tracker (فایل مخفی)

    ls -la /tmp/.pg_state


-----------------------------------------------
2- بررسی سرویس‌های systemd مخرب

بررسی وجود سرویس مخرب pgmon

    systemctl --user list-units --type=service | grep -i pgmon

بررسی وضعیت سرویس

    systemctl --user status pgmon.serv



-----------------------------------------------
۳. بررسی ترافیک شبکه (Network IoCs)

در لاگ‌های فایروال، DNS، و Proxy دنبال این دامنه‌ها بگرد:

بررسی در لاگ‌های DNS

    grep -r "aquasecurtiy" /var/log/
    grep -r "icp0.io" /var/log/

بررسی کانکشن‌های فعال

    ss -tulpn | grep -E "icp0|aquasecurtiy"
    netstat -an | grep -E "icp0|aquasecurtiy"



-----------------------------------------------
4. بررسی DaemonSet های مخرب در صورت داشتن کلاستر کوبرنتیز


لیست DaemonSet ها در namespace کلیدی

    kubectl get daemonset -n kube-system

بررسی مستقیم نام‌های مخرب

    kubectl get daemonset host-provisioner-iran -n kube-system
    kubectl get daemonset host-provisioner-std -n kube-system

اگر هر کدام وجود داشتند، فوری بررسی کن:

بررسی جزئیات DaemonSet مخرب

    kubectl describe daemonset host-provisioner-iran -n kube-system
    kubectl describe daemonset host-provisioner-std -n kube-system

نشانه‌های خطرناک در output:

    privileged: true
    hostPID: true یا hostNetwork: true
    tolerations: [operator: Exists]

## اقدامات لازم در صورت آلوده شدن سیستم

اگر متوجه شدید که سیستم یا کلاستر شما به CanisterWorm آلوده شده است، باید بلافاصله وارد وضعیت «پاسخ به رخداد» (Incident Response) شوید. با توجه به ماهیت خود-تکثیرشونده و مخرب این کرم، سرعت عمل بسیار حیاتی است.


۱. قطع دسترسی و قرنطینه (Isolation)
اولین قدم، جلوگیری از گسترش بیشتر کرم و ارتباط آن با شبکه بلاک‌چین است:

قطع اینترنت: اگر مقدور است، سیستم آلوده را از شبکه قطع کنید.

توقف سرویس مخرب: در ترمینال دستورات زیر را برای از کار انداختن بک‌دور سیستم اجرا کنید:

    Bash
    systemctl --user stop pgmon.service
    systemctl --user disable pgmon.service
    rm -f ~/.config/systemd/user/pgmon.service

پاکسازی پروسه‌ها: با دستور pkill -f service.py مطمئن شوید هیچ نسخه در حال اجرایی از بدافزار باقی نمانده است.

۲. ابطال و تعویض فوری کلیدها (Credential Rotation)
فرض را بر این بگذارید که تمام رمزها و کلیدهای موجود در سیستم به دست مهاجمان رسیده است. بدون تأمل موارد زیر را انجام دهید:

npm: توکن‌های npm را از پنل کاربری ابطال (Revoke) کنید.

GitHub/GitLab: تمامی Personal Access Tokenها و کلیدهای SSH متصل به اکانت را حذف کنید.

K8s: کلیدهای API و سرویس‌اکانت‌های کلاستر را بازنشانی کنید.

۳. پاکسازی کلاستر کوبرنتیز (K8s Remediation)
اگر کلاستر شما آلوده شده، پاک کردن پادها کافی نیست چون DaemonSet دوباره آن‌ها را می‌سازد:

حذف DaemonSet مخرب: ابتدا منبع اصلی را پیدا و حذف کنید:

    kubectl delete ds <نام-مشکوک> -n <نام-نیمپیس>

بررسی نودهای فیزیکی: چون کرم از کانتینر فرار کرده و روی Host OS نشسته است، باید روی تک‌تک نودهای کلاستر (Worker & Master Nodes) مراحل بخش ۱ (توقف سرویس pgmon) را تکرار کنید. در موارد شدید، نصب مجدد سیستم‌عامل نودها (Re-imaging) تنها راه اطمینان ۱۰۰٪ است.

۴. بازرسی پکیج‌های منتشر شده (Audit)
بزرگترین خطر CanisterWorm این است که شما را به مهره‌ای برای آلوده کردن دیگران تبدیل می‌کند:

به حساب npm خود بروید و لیست پکیج‌هایتان را چک کنید.

اگر نسخه‌ی جدیدی (مثلاً یک Patch version مشکوک) بدون اطلاع شما منتشر شده، بلافاصله آن نسخه را Unpublish کنید و به کاربران خود اطلاع‌رسانی کنید.

۵. جلوگیری از آلودگی مجدد (Hardening)
برای اینکه دوباره در دام نیفتید:

استفاده از --ignore-scripts: هنگام نصب پکیج‌های جدید، از دستور npm install --ignore-scripts استفاده کنید تا اسکریپت‌های مخرب postinstall اجرا نشوند.

فعال‌سازی 2FA: برای انتشار پکیج‌ها در npm حتماً احراز هویت دو مرحله‌ای را اجباری کنید.

محدودسازی دسترسی K8s: از Pod Security Standards استفاده کنید تا اجازه اجرای پادهای privileged یا دسترسی به hostPID به هر کسی داده نشود.

## اقدامات پیشگیرانه جهت جلوگیری از آلوده شدن سیستم

### پیشگیری از حملات پیچیده‌ای مثل CanisterWorm که از چندین لایه (npm، سیستم‌عامل و کوبرنتیز) برای نفوذ استفاده می‌کنند، نیازمند یک استراتژی دفاعی چندلایه است.


---------------
۱. ایمن‌سازی فرآیند نصب پکیج‌ها (لایه توسعه‌دهنده)

بیشترین نرخ آلودگی از طریق اجرای خودکار اسکریپت‌های مخرب در زمان نصب رخ می‌دهد.


استفاده از فلگ  --ignore-scripts:   همیشه هنگام نصب پکیج‌های جدید از این دستور استفاده کنید:

    npm install --ignore-scripts

این کار مانع از اجرای اسکریپت‌های postinstall می‌شود که نقطه ورود اصلی CanisterWorm هستند.

استفاده از ابزارهای اسکن لحظه‌ای: ابزارهایی مثل Socket.dev یا Snyk را نصب کنید تا قبل از نصب یک پکیج، رفتارهای مشکوک آن (مثل دسترسی به شبکه یا فایل‌های سیستم) را به شما هشدار دهند.

قفل کردن نسخه‌ها (Lockfiles): همیشه از فایل‌های package-lock.json یا yarn.lock استفاده کنید و اجازه ندهید نسخه‌های پکیج‌ها به صورت خودکار و بدون بررسی شما آپدیت شوند.


---------------------
۲. محدودسازی دسترسی‌ها در سطح سیستم‌عامل
CanisterWorm از دسترسی‌های کاربر جاری برای ساخت سرویس‌های پایداری استفاده می‌کند.

جداسازی محیط توسعه: تا حد امکان پروژه‌های حساس را درون Container (مثل Docker) یا VM توسعه دهید. این کار باعث می‌شود اگر بدافزاری اجرا شد، نتواند به فایل‌های اصلی سیستم شما (مثل کلیدهای SSH) دسترسی پیدا کند.

نظارت بر سرویس‌های کاربر: به صورت دوره‌ای سرویس‌های فعال در سطح کاربر را چک کنید:
systemctl --user list-unit-files --type=service
هر سرویس ناشناسی (مثل pgmon.service) باید بلافاصله بررسی شود.

محافظت از فایل‌های حساس: دسترسی به فایل‌های مهم مثل .npmrc یا .ssh/ را محدود کنید و از نگهداری توکن‌های npm به صورت "Plain Text" خودداری کنید.



---------------------
۳. سخت‌گیرانه کردن امنیت کوبرنتیز (لایه زیرساخت)
برای جلوگیری از فرار بدافزار از کانتینر به نودهای اصلی کلاستر:

اجرای سیاست عدم امتیاز (Least Privilege): از Pod Security Admission استفاده کنید تا اجازه اجرای پادهایی با ویژگی‌های زیر داده نشود:

    privileged: true

    hostPID: true

    hostNetwork: true

محدودسازی دسترسی به API Server: از RBAC (کنترل دسترسی مبتنی بر نقش) استفاده کنید. اجازه ندهید پادهای معمولی یا CI/CD Runnerها توانایی ایجاد DaemonSet یا Deployment در تمام Namespaceها را داشته باشند.

استفاده از Runtime Security: ابزارهایی مثل Falco یا Tetragon را نصب کنید تا به محض اینکه پروسه‌ای سعی کرد فایلی را در سیستم‌عامل میزبان تغییر دهد یا ترافیک مشکوکی به شبکه ICP بفرستد، بلافاصله آن را مسدود کنند.



--------------------
۴. مدیریت هوشمند توکن‌ها و اعتبارنامه‌ها
فعال‌سازی 2FA اجباری: در اکانت npm و GitHub خود، احراز هویت دومرحله‌ای را برای تمام عملیات‌های حساس (مخصوصاً Publish کردن پکیج) فعال کنید.

استفاده از توکن‌های کوتاه‌مدت: به جای استفاده از توکن‌های دائمی در CI/CD، از OIDC (مثل GitHub Actions OIDC) استفاده کنید تا توکن‌ها فقط برای چند دقیقه معتبر باشند.

چرخش خودکار کلیدها (Key Rotation):کلید های دسترسی خود را به صورت خودکار (مثلاً هر ۳۰ روز) تعویض کنید.
